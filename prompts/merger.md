# 增量 Merge 逻辑

## 任务

当用户追加新的原材料或提交纠偏时，将增量信息 merge 进现有的 `life_model.md` 和 `persona.md`。
要求共享统一变更类型字典，保持更新的可追溯性。

## 统一变更类型字典（Change Types）

- **[ADD / 新增]**：发现全新维度的结论，且无旧结论。
- **[ENHANCE / 增强]**：新材料的高/中置信度证据支持旧结论，提升置信度（从★升为★★★）或增加样本池（证据升级）。
- **[CORRECT / 修正]**：基于用户显式纠正（"这不像我"）或压倒性的新事实，推翻并修改原结论。
- **[CONFLICT / 冲突]**：新生成的高置信结论与旧高置信结论打架。保留双方不覆盖，抛出需用户确认的 ⚠️冲突节点。
- **[DOWNGRADE / 降级]**：部分新材料与旧高置信结论轻微矛盾，但不构成推翻，下调其绝对性或置信度（例如：标记“在特定高压下有反常表现”）。

## 原则规范

1. **增量不盲目覆盖**：禁止静默覆盖高置信度旧结论。必须挂载对应变更类型。
2. **强制运用统一证据协议**：新增片段必须挂载 `[E序号|来源|时间]` 。
3. **用户纠正至上**：通过 Correction 发起的变更，类型必须为 **[CORRECT]**，优先级最高。

## 工作流

### 第 1 步：解析与校验合并冲突
按照 `life_analyzer.md` 与 `persona_analyzer.md` 对增量部分提取结论。对比原文件版本，打上字典标签。
对于 [CONFLICT] 类型，必须等待用户确认后才合并。

### 第 2 步：增量文件更新
直接将变更后的结论合并至当前版本的双模型 markdown 文件中，更新带有标签与增量证据：
```markdown
- **效率优先** (★★★)
  *证据链：[旧E1], [新E2|微信工作群|2024-11]*
  *[ENHANCE] 近期工作沟通中此表现进一步强化*
```

### 第 3 步：版本归档与溯源
创建或追加 `CHANGELOG.md` 更新记录记录。

```markdown
=== 增量合并摘要 ===

【ADD/新增】
+ 关系边界 - 对同事：保持表面客气但绝不分享生活 (★★) [E12|微信|2024-10]

【ENHANCE/增强】
↑ 长期主义置信度增强：补充证据 [E13|日记|2024-11]，提升至 (★★★)

【CORRECT/修正】
✓ 纠正决策原则：由“极度保守” 修正为 “对内侧重防御，对外敢于投资”。原因：用户指出旧结论有误。

【CONFLICT/冲突⚠️】
发现新的表达风格出现“极度情绪化”，与原档案记录的“理智温和”冲突。
请确认：这是否是特定压力下的特异点，还是发生了人格转变？
```
